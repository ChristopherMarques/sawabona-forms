# Contribuindo para @sawabona/forms

Primeiro, obrigado por considerar contribuir para o Sawabona Forms! São pessoas como você que fazem desta uma ótima ferramenta.

## Como Posso Contribuir?

### Reportando Bugs

Esta seção guia você sobre como enviar um relatório de bug para o Sawabona Forms. Seguir estas diretrizes ajuda os mantenedores e a comunidade a entender seu relatório, reproduzir o comportamento e encontrar relatórios relacionados.

*   **Use um título claro e descritivo** para a issue para identificar o problema.
*   **Descreva os passos exatos para reproduzir o problema** com o máximo de detalhes possível.
*   **Forneça exemplos específicos para demonstrar os passos**. Inclua links para arquivos ou projetos GitHub, ou snippets de código que você usou nesses exemplos.

### Sugerindo Melhorias

Esta seção guia você sobre como enviar uma sugestão de melhoria para o Sawabona Forms, incluindo funcionalidades completamente novas e pequenas melhorias em funcionalidades existentes.

*   **Use um título claro e descritivo** para a issue para identificar a sugestão.
*   **Forneça uma descrição passo-a-passo da melhoria sugerida** com o máximo de detalhes possível.
*   **Explique por que essa melhoria seria útil** para a maioria dos usuários do Sawabona Forms.

### Pull Requests

O processo descrito aqui tem vários objetivos:

- Manter a qualidade do Sawabona Forms
- Corrigir problemas que são importantes para os usuários
- Engajar a comunidade em trabalhar nas melhores ferramentas possíveis
- Habilitar um sistema sustentável para os mantenedores do Sawabona Forms revisarem contribuições

Por favor, siga estes passos para ter sua contribuição considerada pelos mantenedores:

1.  **Faça um Fork do repositório** e crie sua branch a partir da `main`.
2.  Se você adicionou código que deve ser testado, adicione testes.
3.  Se você mudou APIs, atualize a documentação.
4.  Garanta que a suíte de testes passe.
5.  Garanta que seu código siga o estilo do projeto (lint).
6.  Abra o Pull Request!

## Guias de Estilo

### Mensagens de Commit

*   Use o tempo presente ("Adiciona funcionalidade" e não "Adicionada funcionalidade")
*   Use o modo imperativo ("Move cursor para..." e não "Move o cursor para...")
*   Limite a primeira linha a 72 caracteres ou menos
*   Referencie issues e pull requests liberalmente após a primeira linha

### Guia de Estilo JavaScript/TypeScript

Todo JavaScript/TypeScript deve aderir às regras do Prettier/ESLint configuradas no projeto.

## Notas Adicionais

### Labels de Issues e Pull Requests

Esta seção lista as labels que usamos para nos ajudar a rastrear e gerenciar issues e pull requests.

*   `bug` - Issues que vêm com uma reprodução de erro.
*   `enhancement` - Issues que pedem uma nova funcionalidade.
*   `documentation` - Issues que afetam a documentação.
